Concurrency Scaling
Total Throughput vs Concurrent Requests
vLLM continuous batching vs Ollama sequential serving
Why vLLM scales: At n=4, vLLM batches all 4 requests into a single GPU forward pass per step — matrix × matrix instead of matrix × vector. Same VRAM bandwidth, 4× the output. Effective bytes/token drops from 2,350MB to ~587MB, pushing past the single-request ceiling.
Average Latency (TTFT) vs Concurrent Requests
Time to first token — what the user actually feels
Why Ollama's latency explodes: Ollama serves requests sequentially. At n=4, request 4 waits for requests 1, 2, 3 to fully complete before seeing its first token. 10 seconds TTFT = user has already closed the tab.
llama.cpp — GPU Layer Split Analysis
Generation Speed vs GPU Layers
Same model, same weights — different memory routing
The PCIe penalty: Half-GPU (ngl=16) gives 14 t/s — barely better than pure CPU at 8.4 t/s. Each token requires a round-trip across the PCIe bus between GPU and CPU layers. The bus latency (~75μs × 200 tokens = 15s overhead) dominates.
VRAM Consumption by Configuration
Model weights + KV cache + compute buffers
Memory breakdown @ full GPU: 2,228 MiB model weights + 1,536 MiB KV cache (4K context) + 74 MiB compute = 3,839 MiB total. KV cache is pre-allocated for max context — this is the fragmentation problem vLLM's paged attention solves.
